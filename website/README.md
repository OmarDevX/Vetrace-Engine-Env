# Vetrace website

This directory contains the static Vetrace homepage, searchable examples gallery, live example runner, and generated WebAssembly package.

```bash
./scripts/build_web.sh
./scripts/serve_web.sh
```

Do not edit files inside `website/pkg/`; they are generated by `wasm-pack`.

The live examples use the same `vetrace_render::WgpuRenderer` as desktop WGPU builds. `vetrace_web` only supplies the canvas, DOM input, browser asset preloading, and `requestAnimationFrame` runner.

## Runtime troubleshooting

`304 Not Modified` is normal HTTP caching, but the development server now sends
`Cache-Control: no-store` so stale loaders do not hide fixes. The examples page
shows the exact boot phase and error.

A runnable checkout must contain `website/pkg/vetrace_web.js` and
`website/pkg/vetrace_web_bg.wasm`. Run `./scripts/build_web.sh` to generate them.
`./scripts/serve_web.sh` now builds automatically when the tools are installed,
and refuses to silently serve a website with a missing runtime.

WebGPU must be available in a secure context. For local development use the
printed `http://127.0.0.1:PORT` URL, not a LAN HTTP address or `file://` URL.

### WebGPU limit compatibility

The engine currently uses `wgpu 0.20.1` to preserve the existing renderer API.
Modern browsers removed the old `maxInterStageShaderComponents` device-limit
name. `vetrace_web` installs a narrow compatibility wrapper before WGPU starts
that removes only that retired key. Rebuild `website/pkg/` after updating so the
generated JavaScript includes the wrapper.

### Linux hybrid-GPU compatibility

Browser examples request the low-power adapter by default. This avoids a known
Dawn/Vulkan external-memory failure seen on some Linux discrete-GPU paths. To
explicitly test the dedicated adapter, append `?gpu=high-performance` to the
example URL. Custom pages can set `data-gpu-preference="high-performance"` on
the canvas before constructing `WebRenderer`.
